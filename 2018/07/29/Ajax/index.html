Ajax (全称 ：Asynchronous JavaScript and XML –> 异步的 JS 和 XML)

这一技术可以向服务器请求额外的数据而无需卸载页面，会带来更好的用户体验

  1. 虽然名字中包含‘XML’的成分，但 Ajax 通信与数据格式无关，并不一定要是 xml 格式的数据（名字中会有 xml 是因为早期使用该技术在服务器与客户端传递的数据主要是 xml 格式的而已，而现在传递的主要是 json 格式的数据）
    2.名字中的异步，指的主要是特指局部刷新，而非同步或异步这样的概念。

一些细节知识点

全局刷新

尽管当前的页面只需要更新部分的数据，但服务器也会把整个页面的数据重新渲染后返回。最后客户端需要刷新页面才能看到最终的结果。

非前后端完全分离（数据渲染在服务器端）

前端开发只需要完成页面的制作，并且把页面中的一些交互效果完成即可。页面中需要动态呈现的部分，都是交给后端开发人员进行数据绑定与基于服务器端的数据渲染。

优点：
1.服务器返回客户端的源代码中是可以看到动态数据的，这样有利于 SEO 优化；
2.由于数据渲染在服务器端，客户端最后只需要渲染代码就可以，不需要额外的处理了。所有，如果服务器的计算速度快的话，页面的呈现速度也会较快。
缺点：
1.不利于团队协作开发，开发效率较低;
2.服务器的压力较大。

前后端完全分离（数据渲染在客户端）

页面中需要动态绑定获得的数据，是由客户端完成的。
1.向服务器端发送 Ajax 请求；
2.把得到的数据解析为对象字符串以后，拼接得到 HTML 字符串；
3.把拼接好的 HTML 片段插入到合适的位置，不需要刷新整个页面。

优点：
1.在接口文档完成后，前后端人员就可以分头进行各自的开发了。这样有利于加快开发效率，真正意义上实现了前后端技术的区分；
2.可以很方便的请求一些实时的数据，而不需要每次都重新刷新整个页面，会让用户体验更好。
缺点：
1.在源代码中，看不到动态绑定的数据，不利于 SEO 优化；
2.加载页面时，大致上是：第一次向服务器发送请求，请求到的结果中只有页面的 HTML 和 CSS，还没有数据。第二次向服务器发送请求，将请求到的数据加入到页面相应的部分…过程比非前后端完全分离要繁琐一些，因此，页面的加载速度可能要慢一些。

URL、URI、URN

URL：统一资源定位符
URI：统一资源标识符
URN：统一资源名称
URI = URL + URN

Ajax 中的请求方式

GET 系列请求

  1. get：从服务器获取资源
    2.head：获取响应的头部信息
    3.delete：删除服务器上的指定的资源文件
  2. ……
    POST 系列请求
    1.post：向服务器推送数据
    2.put：向服务器推送文件资源（数据量比post的要大）

GET 系列请求和 POST 系列请求在实际的使用中，存在着很多的区别

1.数据量大小问题
GET 系列请求传递给服务器的数据内容没有 POST 系列请求传递的多。
原因：GET 请求传递给服务器的内容一般都基于URL地址问号传参来实现的，而 POST 请求一般都是基于设置请求主体来实现的。
各浏览器都有自己的关于 URL 地址最大长度的限制（谷歌：8KB，火狐：7KB，IE：2KB）。超出限制长度的部分，浏览器会自动截取掉，导致传输给服务器的数据缺失。
理论上，POST通过请求主体传送给服务器的数据是没有大小限制的。但是，在真实项目中，为了保证传输速率，会限制上传数据量的大小。

2.安全性问题
GET 系列请求的安全性没有 POST 系列请求的安全性高（虽然 POST 的安全性也不是特别的高）
原因：GET 请求是通过URL 地址传参的，有‘URL劫持’的风险，容易泄露数据。而POST 请求传递的数据是设置在请求主体里的，黑客要获得数据，就要麻烦一些。

3.缓存问题
GET 系列请求可能会出现缓存（缓存不可控，一般不使用）。而 POST 请求就不会产生缓存（除非自己进行特殊处理）。
原因：GET 请求是通过 URL 地址问号传参的，而 POST 请求是通过请求主体传参的。设置请求主体不会出现缓存，而 URL传递参数就会出现缓存了。

拓展问题

当在浏览器地址栏中输入一个URL地址，到最后看到页面，中间都经历了哪些事情？

Request 请求阶段
1.首先根据客户端输入的域名，到DNS服务器上进行反解析（通过域名找到服务器的外网 IP）
2.通过找到的外网 IP，找到对应的服务器
3.通过在地址栏找到的端口号（没输入是因为不同的协议有自己的默认的端口号）找到服务器上发布的对应的项目

Response 响应阶段
4.服务器获取到请求资源文件的地址，把资源文件中的源代码找到
5.服务器会把找到的源代码返回给客户端

浏览器自主渲染阶段
6.客户端接收到源代码后，会交到浏览器的内核（渲染引擎）进行渲染，最后由浏览器绘制出页面
